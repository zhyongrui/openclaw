# Project Blueprint Schema

## Fixed Path

The canonical project blueprint path is:

`PROJECT-BLUEPRINT.md`

It lives at the repository root so it can be committed, reviewed, and reused
across sessions and operator hosts.

## Lifecycle Statuses

The blueprint frontmatter currently supports these statuses:

- `draft`
- `clarified`
- `agreed`
- `active`
- `superseded`

`openclaw code blueprint-set-status --status agreed` is the first explicit
agreement checkpoint for the blueprint-first control plane.

## Current Scaffold

The scaffold generated by `openclaw code blueprint-init` contains:

- YAML frontmatter with:
  - `schemaVersion`
  - `title`
  - `status`
  - `createdAt`
  - `updatedAt`
  - `statusChangedAt`
  - `agreedAt` once the blueprint is explicitly agreed
- required sections:
  - `Goal`
  - `Success Criteria`
  - `Scope`
  - `Non-Goals`
  - `Constraints`
  - `Risks`
  - `Assumptions`
  - `Human Gates`
  - `Provider Strategy`
  - `Workstreams`
  - `Open Questions`

## Current CLI Surface

- `openclaw code blueprint-init`
  - create the fixed blueprint scaffold
- `openclaw code blueprint-show`
  - inspect the current blueprint state, including revision metadata
- `openclaw code blueprint-clarify`
  - surface deterministic clarification questions and suggestions from the
    current blueprint contents
- `openclaw code blueprint-set-status`
  - record lifecycle transitions such as `agreed`
- `openclaw code blueprint-decompose`
  - derive and persist `.openclawcode/work-items.json` from the current
    blueprint
- `openclaw code work-items-show`
  - inspect the current repo-local work-item inventory artifact and stale-state
    signals

The machine-readable blueprint summary currently includes:

- lifecycle state:
  - `status`
  - `agreedAt`
  - `statusChangedAt`
- revision metadata:
  - `revisionId`
  - `contentSha256`
  - `frontmatterKeys`
- readiness metadata:
  - `defaultedSections`
  - `defaultedSectionCount`
  - `workstreamCandidateCount`
  - `openQuestionCount`
  - `humanGateCount`
  - `providerRoleAssignments`

The first repo-local work-item artifact now lives at:

`/repo/.openclawcode/work-items.json`

It currently carries:

- blueprint linkage:
  - `blueprintPath`
  - `blueprintStatus`
  - `blueprintRevisionId`
  - `currentBlueprintRevisionId`
  - `artifactStale`
- execution gating:
  - `readyForExecution`
  - `readyForIssueProjection`
  - `blockers`
  - `suggestions`
- planned work items:
  - stable ids
  - source section and workstream index
  - provider-role assignments
  - acceptance criteria
  - GitHub issue drafts

The first repo-local discovery artifact now lives at:

`/repo/.openclawcode/discovery-work-items.json`

It currently carries:

- stable discovery evidence ids and dedupe keys
- severity and priority scoring
- discovered work-item drafts for:
  - missing work-item artifacts
  - stale work-item artifacts
  - remaining blueprint open questions

The first provider-neutral role routing artifact now lives at:

`/repo/.openclawcode/role-routing.json`

It currently carries:

- first-class roles:
  - `planner`
  - `coder`
  - `reviewer`
  - `verifier`
  - `docWriter`
- normalized adapter ids:
  - `codex`
  - `claude-code`
  - `openclaw-default`
  - `custom`
- route source, mixed-mode detection, unresolved-role counts, and fallback-chain
  visibility

All eight commands support `--json` so the blueprint state is already
machine-readable enough for follow-on automation and staged handoff.
